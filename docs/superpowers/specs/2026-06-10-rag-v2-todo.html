簇 1 · LLM 抽象层重构

1.1 评估 langchain 价值 vs 依赖成本

CLUSTER 1 · 01 P2 3d

目标

定量评估 langchain BaseChatModel 抽象的真实价值（Langfuse callback、retry、tool use 抽象），与依赖成本（包体积、版本冲突、抽象泄露）。结论二元：保留（≥3 个不可替代能力）或砍改直接用 anthropic SDK。

触发条件

  • 启动评估：python -m app.eval.langchain_audit
  • 输出报告：保留/砍改 + 影响面

API 改动

评估阶段不改动；决策落地后：

  • 砍改路径：删除 langchain-anthropic 依赖，app/llm/factory.py 重写为直接 anthropic.AsyncAnthropic + 自写 callback
  • 保留路径：无代码改动，仅生成 ADR

数据流

app.eval.langchain_audit
  → 静态分析：grep langchain 实际使用 API
  → 依赖成本：pip show langchain langchain-anthropic langchain-core
  → 价值评估表（1-5 分）：
      Langfuse callback 集成          5  ← 不可替代
      retry / streaming 包装          3
      tool use 抽象                   4  ← 跟 langgraph 绑
      ChatAnthropic vs 直接 anthropic 2
  → 决策：总分 ≥ 12 → 保留；< 12 → 砍改（1 周迁移窗口）

测试要点

  • 静态：所有 from langchain_* import 列入清单
  • 基准：cold start time / RSS / token 计数精度
  • 灰度：5% 流量跑直接 SDK，95% langchain，1 周后切 100%

依赖

v1 LLM 工厂已稳态运行。v1 依赖：langchain==1.0.8、langchain-core>=1.0.8、langchain-anthropic>=0.3

与 v1 的边界

v1：保留 langchain（v0.4 决策 1）。v2：重新评估，结论独立。

风险

  • 砍改后回调链断（Langfuse 漏报）→ 灰度 + 对照埋点
  • 砍改后 token 计数偏差 → RAGAS eval 必须绿

验收

  • [ ] 评估报告落 ADR
  • [ ] 决策落地后 CI 全绿
  • [ ] Langfuse trace 无回归

1.2 LangGraph init_chat_model 配置驱动

CLUSTER 1 · 02 P2 2d

目标

把 v1 写死的 make_llm() 工厂换成 langgraph 推荐的 init_chat_model 配置驱动模式：模型 provider、temperature、max_tokens、model_name 全部走 config.yaml / 环境变量，不改代码即可切换。

触发条件

  • 部署时切换模型
  • 灰度：10% 流量跑新模型
  • A/B 实验（与簇 6.1 联动）

API 改动

app/llm/factory.py 重构：

from langchain.chat_models import init_chat_model

def make_llm(node: str) -> BaseChatModel:
    cfg = load_config(f"llm.{node}")
    return init_chat_model(
        model=cfg.model,
        temperature=cfg.temperature,
        max_tokens=cfg.max_tokens,
        configurable_fields=("model", "temperature"),
    )

数据流

节点调用 make_llm("answer") → 读 config/llm.yaml → init_chat_model(...) 返回 BaseChatModel → 注入 Langfuse callback。切模型 = 改 yaml + 重启服务（不热更）。

测试要点

  • 单元：config 加载、缺字段报错
  • 集成：4 节点各跑一条 query，Langfuse trace 标注 model_used
  • 灰度：单节点切模型不影响其他节点

依赖

langgraph==1.0.5 已具备 init_chat_model。1.1 决策落地（如果 1.1 砍改 langchain，本项需重写）。

与 v1 的边界

v1：模型写死在 make_llm()。v2：配置驱动，不做热更。

风险

  • 改配置忘记重启 → 加 startup check
  • 配错字段 → config schema 用 Pydantic 强校验

验收

  • [ ] 改 yaml 切模型无需改代码
  • [ ] 4 节点可独立配模型
  • [ ] 启动 fail-fast

簇 2 · 多模态

2.1 多模态 VLM（CLIP / Claude vision）接入

CLUSTER 2 · 01 P1 1w

目标

v1 仅索引图片元数据 + 缩略图（不索引内容）。v2 接入 VLM：图片入库时调 VLM 生成 caption，caption 入索引参与检索。

触发条件

  • ingest 时检测到图片元素（LlamaIndex ImageElement）
  • 自动调 VLM caption

API 改动

  • 检测图片元素后调 VLMClient.caption(image_bytes) → string
  • caption 走与正文相同的 splitter（短，~50 token）
  • metadata 增 image_ref、vlm_caption_id、vlm_model

数据流

图片元素 (image_bytes, ocr_text?)
  → sha256 → artifacts/{doc_id}/images/{hash}.png
  → VLMClient.caption(image_bytes)
  → SentenceSplitter → embed → upsert
  → metadata: { image_ref, vlm_model, vlm_caption_id, ocr_text_ref? }

测试要点

  • 单元：VLM 客户端 mock
  • 集成：含 5 张图的 PDF，索引后查询"图1的标题"能命中 caption chunk
  • 成本：VLM caption 单图 ~$0.001

依赖

不依赖其他 v2 项。外部：VLM provider（Claude vision / OpenAI GPT-4V / 本地 Qwen-VL）。v1 基础：langchain==1.0.8、llama-index==0.14.8

与 v1 的边界

v1：图片不索引内容。v2：索引 caption + OCR，不做"以图搜图"。

风险

  • VLM 幻觉 → RAGAS 评测加 image_faithfulness
  • 成本放大 → 配置化开关，默认关
  • 隐私 → 本地 Qwen-VL 走自部署

验收

  • [ ] caption 落索引，可检索
  • [ ] 成本可控
  • [ ] RAGAS image_faithfulness ≥ 0.6

簇 3 · 检索质量

3.1 hybrid search (BM25 + dense)

CLUSTER 3 · 01 P0 4d

目标

v1 纯 dense（向量）检索，缺关键词命中能力。v2 引入 BM25 稀疏 + dense 稠密双路，加权融合（RRF 或线性加权）。

数据流

retrieve_node(state)
  → for r in rewrites:
        sparse_hits = bm25_search(r, k=20)
        dense_hits  = knn_search(r, k=20)
        fused = rrf(sparse_hits, dense_hits) → top 20
        chunks += fused
  → dedupe(chunks)

测试要点

  • golden set 50 题，hybrid vs pure dense：context_precision ≥ +10%, latency < dense × 1.5

依赖

OpenSearch 2.19 已部署；索引重建（加 BM25 字段）。v1 基础：opensearch-py>=2.6,<3

验收

  • [ ] RAGAS context_precision 提升 ≥ 10%
  • [ ] P95 latency 增长 < 50%

3.2 专用 reranker（bge-reranker / Cohere）

CLUSTER 3 · 02 P0 3d

目标

v1 rerank 用 LLM 打分 0-10，慢 + 贵。v2 切专用 rerank 模型，单 query 20-50ms，$0.001/call。

数据流

chunks: list[Chunk] (top 20 from retrieve)
  → bge_reranker.score(query, [c.text for c in chunks])
  → chunks_sorted = sort by score desc
  → return top 5

验收

  • [ ] 质量不退
  • [ ] latency/cost 大幅降
  • [ ] fallback 路径测过

簇 4 · 流式与性能

4.1 流式响应 SSE

CLUSTER 4 · 01 P0 1w

目标

POST /api/chat v1 等 LLM 全答完才返回（3-5s）。v2 改 SSE：LLM 首个 token 即可流到 Gradio，UX 提升 2-3 倍感知速度。

API 改动

  • StreamingResponse(media_type="text/event-stream")
  • 事件类型：event: sources / event: token / event: done

数据流

StreamingResponse 生成器:
  for event in graph.astream_events(input, config, version="v2"):
      if event["event"] == "on_retriever_end":
          yield { sources }
      elif event["event"] == "on_chat_model_stream":
          yield { token: ... }
      elif ... "on_chain_end" and name=="answer":
          yield { done, session_id }
  save_memory 异步（在 done 后 fire-and-forget）

验收

  • [ ] Gradio 看到 token 滚动
  • [ ] 中断不污染 DB

7.2 Snowball 增量同步

CLUSTER 7 · 02 P2 3d

目标

任意 URL 源（internal wiki / Notion / GitBook）的"按图索骥"增量抓取：从 seed URL 出发，BFS 发现新 URL 入队。

数据流

每日 02:00
  → frontier = [seed]
  → visited = set()
  → for depth in 0..3:
        for url in frontier:
            if url in visited: continue
            fetch → parse → upsert
            visited.add(url)
            for link in extract_links(html):
                if link_domain in allowed and link not in visited:
                    new_pages.append(link)
        frontier = new_pages
        if len(visited) > max_pages_per_run: break

验收

  • [ ] BFS 正确
  • [ ] 限速不爆
