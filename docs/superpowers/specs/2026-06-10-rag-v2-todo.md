## 1. 簇 1 · LLM 抽象层重构

### 1.1 评估 langchain 价值 vs 依赖成本，决定是否砍改 anthropic SDK 直调

#### 目标

定量评估 langchain `BaseChatModel` 抽象在 RAG 系统里的真实价值（Langfuse callback、retry、tool use 抽象），与依赖成本（包体积、版本冲突、抽象泄露）。结论二元：保留 langchain（提供 ≥3 个不可替代能力）或砍改直接用 anthropic SDK。

#### 触发条件

- 启动评估：`python -m app.eval.langchain_audit`
- 输出报告：保留/砍改 + 影响面

#### API 改动

评估阶段不改动；决策落地后：
- 砍改路径：删除 `langchain-anthropic` 依赖，`app/llm/factory.py` 重写为直接 `anthropic.AsyncAnthropic` + 自写 callback
- 保留路径：无代码改动，仅生成 ADR

#### 数据流

```
app.eval.langchain_audit
  → 静态分析：
      grep langchain 实际使用 API
      列出：ChatAnthropic、BaseChatModel、callbacks、tool use
  → 依赖成本：
      pip show langchain langchain-anthropic langchain-core
      dist 大小、import time（cold start 测量）
  → 价值评估表（1-5 分）：
      Langfuse callback 集成          5  ← 不可替代
      retry / streaming 包装          3
      tool use 抽象                   4  ← 跟 langgraph 绑
      ChatAnthropic vs 直接 anthropic 2
  → 决策：总分 ≥ 12 → 保留；< 12 → 砍改（1 周迁移窗口）
```

#### 测试要点

- 静态：所有 `from langchain_* import` 列入清单
- 基准：cold start time / RSS / token 计数精度
- 灰度：决策"砍改"后 5% 流量跑直接 SDK，95% langchain，1 周后切 100%

#### 依赖

- v1 LLM 工厂已稳态运行
- v1 依赖：langchain==1.0.8、langchain-core>=1.0.8、langchain-anthropic>=0.3

#### 与 v1 的边界

- v1：保留 langchain（v0.4 决策 1）
- v2：重新评估，结论独立

#### 风险

- 砍改后回调链断（Langfuse 漏报）→ 灰度 + 对照埋点
- 砍改后 token 计数偏差 → RAGAS eval 必须绿

#### 验收

- [ ] 评估报告落 ADR
- [ ] 决策落地后 CI 全绿
- [ ] Langfuse trace 无回归（callback path 等价）

### 1.2 LangGraph `init_chat_model` 配置驱动

#### 目标

把 v1 写死的 `make_llm()` 工厂换成 langgraph 推荐的 `init_chat_model` 配置驱动模式：模型 provider、temperature、max_tokens、model_name 全部走 `config.yaml` / 环境变量，**不**改代码即可切换。

#### 触发条件

- 部署时切换模型（如测试用 MiniMax-M3，生产用 Claude Sonnet）
- 灰度：10% 流量跑新模型
- A/B 实验（与簇 6.1 联动）

#### API 改动

`app/llm/factory.py` 重构：
```python
from langchain.chat_models import init_chat_model

def make_llm(node: str) -> BaseChatModel:
    cfg = load_config(f"llm.{node}")  # classify / rewrite / rerank / answer
    return init_chat_model(
        model=cfg.model,                # "anthropic:claude-sonnet-4"
        temperature=cfg.temperature,
        max_tokens=cfg.max_tokens,
        configurable_fields=("model", "temperature"),
    )
```

`config/llm.yaml`：
```yaml
llm:
  classify:
    model: "anthropic:claude-haiku-3"
    temperature: 0.0
  rewrite:
    model: "anthropic:claude-haiku-3"
  rerank:
    model: "anthropic:claude-sonnet-4"
  answer:
    model: "anthropic:claude-sonnet-4"
    temperature: 0.2
```

#### 数据流

```
节点调用 make_llm("answer")
  → 读 config/llm.yaml
  → init_chat_model(...) 返回 BaseChatModel
  → 注入 Langfuse callback（与 v1 相同）
  → 节点 invoke 时按 config 走

切模型：
  修改 config/llm.yaml.answer.model
  → 重启服务（不热更，避免长任务中断）
```

#### 测试要点

- 单元：config 加载、缺字段报错
- 集成：4 个节点各跑一条 query，Langfuse trace 标注 `model_used`
- 灰度：单节点切模型不影响其他节点

#### 依赖

- langgraph 升级到支持 `init_chat_model` 的版本（langgraph==1.0.5 已具备）
- 1.1 决策落地（如果 1.1 砍改 langchain，本项需重写）

#### 与 v1 的边界

- v1：模型写死在 `make_llm()`
- v2：配置驱动，**不**做热更

#### 风险

- 改配置忘记重启 → 加 startup check：节点 init 时打 log "model=X temperature=Y"
- 配错字段 → config schema 用 Pydantic 强校验

#### 验收

- [ ] 改 yaml 切模型无需改代码
- [ ] 4 节点可独立配模型
- [ ] 启动 fail-fast（配错立即报错）

## 2. 簇 2 · 多模态

### 2.1 多模态 VLM（CLIP / Claude vision）接入

#### 目标

v1 仅索引图片**元数据** + 缩略图（不索引内容）。v2 接入 VLM：图片入库时调 VLM 生成 caption，caption 入索引参与检索；用户查询"那个蓝色图表"也能命中。

#### 触发条件

- ingest 时检测到图片元素（LlamaIndex `ImageElement`）
- 自动调 VLM caption

#### API 改动

`app/ingest/pipeline.py`：
- 检测图片元素后，调 `VLMClient.caption(image_bytes)` → string
- caption 走与正文相同的 splitter（短，~50 token，直接做 chunk）
- metadata 增 `image_ref`、`vlm_caption_id`、`vlm_model`

`app/llm/factory.py`：新增 `make_vlm()`。

#### 数据流

```
图片元素 (image_bytes, ocr_text?)
  → 计算 sha256 → artifacts/{doc_id}/images/{hash}.png 落盘
  → VLMClient.caption(image_bytes, prompt="简短描述这张图")
  → caption_text = strip + normalize
  → SentenceSplitter(caption_text)  # 短，可能就 1 chunk
  → embed(caption_text) → upsert
  → 同时把 ocr_text（如有）单独 chunk
  → metadata: { image_ref, vlm_model, vlm_caption_id, ocr_text_ref? }
```

#### 测试要点

- 单元：VLM 客户端 mock（caption 返回固定字符串）
- 集成：含 5 张图的 PDF，索引后查询"图1的标题"能命中 caption chunk
- 成本：VLM caption 单图 ~$0.001，1k 张图 ~$1

#### 依赖

- 不依赖其他 v2 项
- 外部：VLM provider（Claude vision / OpenAI GPT-4V / 本地 Qwen-VL）
- v1 基础：langchain==1.0.8、llama-index==0.14.8

#### 与 v1 的边界

- v1：图片**不**索引内容
- v2：索引 caption + OCR，**不**做"以图搜图"（embed image embedding 是另一条路，V2.x）

#### 风险

- VLM 幻觉：caption 与图不符 → RAGAS 评测加 `image_faithfulness` 子项
- 成本放大：1 万张图 = $10 → 配置化开关，默认关
- 隐私：图含敏感信息 → VLM provider 选择影响；本地 Qwen-VL 走自部署

#### 验收

- [ ] caption 落索引，可检索
- [ ] 成本可控（开关 + 限速）
- [ ] RAGAS image_faithfulness ≥ 0.6

### 2.2 图片内容问答（image_ref 召回 → VLM 答图）

#### 目标

v2.1 仅索引 caption（v2 行为）；v2.2 在 answer 节点拿到 source 中含 `image_ref` 的 chunk 时，把图片 inline 到 prompt，让 VLM 基于**实际图**答图（"对比这两张图"）。

#### 触发条件

- 检索结果含 `image_ref`
- 用户 query 含"图 / 看 / 对比"等视觉意图词

#### API 改动

`app/graph/nodes.py → answer_node`：
- 接收 `state["chunks"]`
- 若 chunks 含 image_ref，构造多模态 prompt

#### 数据流

```
answer 节点
  → chunks = dedupe + rerank(top-5)
  → image_chunks = [c for c in chunks if c.metadata.image_ref]
  → if image_chunks:
        拉图（OpenSearch 返回 image_url 或本地 artifacts/）
        构造多模态 content: [text, image_1, image_2, ...]
        llm_vision.invoke(...)
    else:
        走 v1 纯文本路径
  → answer_text + image_citations
```

#### 测试要点

- 单元：多模态 content 构造正确
- 集成：query"对比这 2 张图" → 答图能区分

#### 依赖

- 2.1（VLM 接入）必先合

#### 与 v1 的边界

- v1：图片仅做缩略图展示
- v2.2：图片作为"答图证据"

#### 风险

- 答图失败（VLM 不支持）→ 降级回 caption 文本
- 隐私：原图离开本机 → 默认走 VLM API 接受；V3 走本地 VLM 解决

#### 验收

- [ ] 答图有图有真相
- [ ] 失败降级不阻塞

### 2.3 表格结构化抽取 + 表格 embedding 增强

#### 目标

v1 的表格元素只走"纯文本 dump" → chunk。v2 用 LLM 把表格转 Markdown/JSON 结构化，再 embed "整张表"作为额外 chunk（保留表头-单元格关系）。

#### 触发条件

- 解析时识别 TableElement
- 表格行数 < 100（更大表分块）

#### API 改动

`app/ingest/splitter.py`：
- 新增 `TableSplitter`：调 LLM `html→markdown` + 输出 `summary` 一句话
- 输出 3 个 chunk：原始文本 / Markdown / summary

#### 数据流

```
TableElement (html)
  → LLM table_to_markdown(html) → md
  → LLM table_summarize(md) → summary_text
  → embed(md) → chunk_md
  → embed(summary_text) → chunk_summary
  → embed(original_text) → chunk_text (v1 行为)
  → metadata: { table_id, is_table_struct: true, columns: [...] }
```

#### 测试要点

- 单元：表格 HTML 多样性（合并单元格 / 空单元格 / 嵌套）
- 集成：查"Q3 收入"能召回到对应表格的 chunk_md

#### 依赖

- 不依赖其他 v2 项；复用 1.2 LLM 工厂
- v1 基础：langchain==1.0.8

#### 与 v1 的边界

- v1：表格当文本
- v2：3 维索引（原始/MD/summary）

#### 风险

- LLM 抽错（幻觉单元）→ RAGAS 表格子集评测
- 成本：每张表 ~$0.005

#### 验收

- [ ] 表格查得回
- [ ] 表格答案不幻觉

## 3. 簇 3 · 检索质量

### 3.1 hybrid search (BM25 + dense)

#### 目标

v1 纯 dense（向量）检索，缺关键词命中能力。v2 引入 BM25 稀疏 + dense 稠密双路，加权融合（Reciprocal Rank Fusion 或线性加权）。

#### 触发条件

- 所有 `retrieve` 节点
- 评测覆盖"含专有名词 / 代码标识符 / 缩写"query

#### API 改动

`app/retrieval/retriever.py` 重构：
- OpenSearch 同索引加 `text` 字段的 `BM25` 索引（OpenSearch 原生）
- retrieve 流程走 RRF 融合

#### 数据流

```
retrieve_node(state)
  → query_rewrites = state["rewrites"]   # 1-3 句
  → for r in rewrites:
        sparse_hits = bm25_search(r, k=20)
        dense_hits  = knn_search(r, k=20)
        fused = rrf(sparse_hits, dense_hits) → top 20
        chunks += fused
  → dedupe(chunks)
  → return chunks
```

#### 测试要点

- 单元：RRF 加权正确
- 集成：golden set 50 题，hybrid vs pure dense：
  - context_precision: hybrid ≥ +10%
  - latency: hybrid < dense × 1.5

#### 依赖

- OpenSearch 2.19 已部署
- 索引重建（加 BM25 字段）
- v1 基础：opensearch-py>=2.6,<3

#### 与 v1 的边界

- v1：纯 dense
- v2：hybrid，**不**做多模态 hybrid

#### 风险

- 索引翻倍（BM25 字段）→ 接受
- RRF 调参 → A/B 实验（与 6.1 联动）

#### 验收

- [ ] RAGAS context_precision 提升 ≥ 10%
- [ ] P95 latency 增长 < 50%

### 3.2 Cohere Rerank / bge-reranker 替代 LLM rerank

#### 目标

v1 的 rerank 节点用 LLM 打分 0-10，**慢 + 贵**。v2 切专用 rerank 模型（Cohere Rerank 3 / bge-reranker-v2-m3），单 query 20-50ms，$0.001/call。

#### 触发条件

- 所有 rerank 节点
- 预算：~1000 query/天 × $0.001 = $1/天

#### API 改动

`app/graph/nodes.py → rerank_node`：
- 默认走 `BgeReranker`（本地，TEI 已部署）
- 备选 `CohereRerank`（API）
- LLM rerank 保留为 fallback（config 切）

#### 数据流

```
chunks: list[Chunk] (top 20 from retrieve)
  → bge_reranker.score(query, [c.text for c in chunks])
  → chunks_sorted = sort by score desc
  → return top 5
```

TEI 启 rerank 端口（与 embed 端口共存），dockerfile 加 `--max-client-batch-size 32`。

#### 测试要点

- 集成：golden set 上 bge-reranker vs LLM rerank：
  - context_precision 相当或更好
  - latency P95 < 100ms（vs LLM 3s）
  - cost 降 ≥ 80%

#### 依赖

- 1.2（LLM 工厂配置驱动）可并行
- TEI 升级支持 rerank
- v1 基础：tei 容器（v1 已部署）

#### 与 v1 的边界

- v1：LLM rerank
- v2：bge-reranker 默认，LLM 备选

#### 风险

- bge-reranker 偶发 bad case → 保留 LLM fallback

#### 验收

- [ ] 质量不退
- [ ] latency/cost 大幅降
- [ ] fallback 路径测过

## 4. 簇 4 · 流式与性能

### 4.1 流式响应 SSE

#### 目标

`POST /api/chat` v1 等 LLM 全答完才返回（典型 3-5s）。v2 改 SSE：LLM 首个 token 即可流到 Gradio，UX 提升 2-3 倍感知速度。

#### 触发条件

- `POST /api/chat`（自动）
- Gradio `gr.ChatInterface(stream=True)`

#### API 改动

`app/api/chat.py`：
- 改 `StreamingResponse(media_type="text/event-stream")`
- 事件类型：
  - `event: sources` data: JSON（首帧）
  - `event: token` data: 文本片段
  - `event: done` data: `{session_id, total_tokens, duration_ms}`

LangGraph `astream_events(version="v2")`：监听 `on_chat_model_stream`。

#### 数据流

```
POST /api/chat
  → 鉴权
  → 准备 graph.invoke stream
  → StreamingResponse 生成器:
      for event in graph.astream_events(input, config, version="v2"):
          if event["event"] == "on_retriever_end":
              yield { sources }
          elif event["event"] == "on_chat_model_stream":
              yield { token: event["data"]["chunk"].content }
          elif event["event"] == "on_chain_end" and name=="answer":
              yield { done, session_id }
      save_memory 异步（在 done 后 fire-and-forget）
```

#### 测试要点

- 单元：SSE 事件序列化、断流（client disconnect）
- 集成：Gradio 实时显示 token，token 间隔 < 200ms
- 失败：流中出错 → `event: error` + partial answer

#### 依赖

- 不依赖其他 v2 项
- FastAPI 升级（>=0.115 已有 StreamingResponse）
- v1 基础：fastapi>=0.115、langgraph==1.0.5

#### 与 v1 的边界

- v1：全量返回
- v2：SSE 流式

#### 风险

- 流中断 → answer 完整性兜底（save_memory 在 done 后必跑）
- Langfuse 完整 trace 在流结束后才落（用户关浏览器丢 trace）→ 接受

#### 验收

- [ ] Gradio 看到 token 滚动
- [ ] 中断不污染 DB

## 5. 簇 5 · 多租户与权限

### 5.1 多租户 + 组织

#### 目标

v1 单租户，所有用户共享 doc。v2 引入 `org` 维度：org 隔离 doc、session、配额；用户属于 ≥1 org。

#### 触发条件

- 用户注册时强制选 org（个人 org 自动创建）
- 管理员可创建/合并 org

#### API 改动

新增表 `orgs (id, name, plan, created_at)`、`org_members (org_id, user_id, role)`。

`users` 加 `current_org_id`（会话默认 org）。

所有 `ingest_jobs`、`chat_sessions`、`chunks`（OpenSearch 加 `org_id` 字段）加 `org_id` 外键。

**query 改造**：retrieve 强制 `filter: { term: { org_id: current_org_id } }`，跨 org 严格隔离。

#### 数据流

```
POST /api/ingest
  → 鉴权 → 解析 current_user + current_org
  → 写 ingest_jobs.org_id = current_org_id
  → OpenSearch upsert 携带 metadata.org_id

POST /api/chat
  → 鉴权 → 解析 current_org
  → retrieve: filter org_id
  → answer: 引用 source 都标 org_id
```

#### 风险

- 性能：filter 字段加索引

### 5.2 doc 级 ACL

#### 目标

org 隔离之上，doc 可独立设 ACL（"只允许 org 内 X 角色访问" / "全员可见" / "私有不共享"）。

#### 触发条件

- ingest 时附 ACL metadata
- UI 可见 doc 列表 = ACL 过滤后

#### API 改动

OpenSearch `chunks` 加 `acl_read TEXT[]` 字段（用户/角色 id 数组）。

retrieve 节点 filter：
```
{
  "bool": {
    "should": [
      { "term": { "acl_read": "all" } },
      { "terms": { "acl_read": [user_id, user_role1, user_role2, org_id] } }
    ]
  }
}
```

新增 `docs` 表存 doc 元数据与 ACL（不存 chunk 文本，chunk 在 OS）。

#### 数据流

```
POST /api/ingest
  → body.acl_read = ["all"] | ["role:eng"] | ["user:42", "user:43"]
  → upsert chunk: metadata.acl_read = body.acl_read

POST /api/chat
  → 鉴权 → 拿到 user_id, roles, org_id
  → retrieve filter: 上述 should
  → 0 命中 → "未找到相关文档"（不区分"无 doc" vs "无权限"）
```

#### 测试要点

- 单元：ACL 过滤矩阵（4 类用户 × 4 类 doc）
- 集成：用户只能看到自己的 doc
- 越权：返回 0 命中（不暴露 doc 存在性）

#### 依赖

- 5.1（org 隔离）必先合
- v1 基础：opensearch-py>=2.6,<3

#### 与 v1 的边界

- v1：无 doc ACL，所有鉴权用户见所有 doc
- v2：ACL 粒度到 doc

#### 风险

- 越权信息泄露：详细错误会暴露 doc 存在性 → 统一 0 命中兜底
- 性能：acl_read 用 keyword 数组，OpenSearch 原生支持

#### 验收

- [ ] ACL 矩阵测试全过
- [ ] 越权不泄露

### 5.5 设备指纹 + 异地登录告警

#### 目标

异常登录检测：首次设备 / 异地 IP / 高频失败 → 邮件告警 + 临时锁定。

#### 触发条件

- 登录成功
- 失败次数超阈

#### API 改动

新增 `login_events (id, user_id, ip, ua, fingerprint, success, created_at)`。

`app/auth/anomaly.py`：
- 指纹：UA + IP 前 24 位 + accept-language hash
- 异地：GeoIP（MaxMind GeoLite2）
- 规则：与最近 30 天登录比对，>500km 且 <30min 视为异地

告警通道：邮件（与 5.4 邮件共用 provider）。

#### 数据流

```
POST /api/auth/login
  → 鉴权成功
  → 查 login_events WHERE user_id, last 30 days
  → 计算新登录 fingerprint
  → if fingerprint in known_devices: ok
    else: mark new_device
  → 查最近一次 login location
  → if distance > 500km AND time_diff < 30min: anomaly!
  → 发邮件告警（"新设备 + 异地登录"）
  → 写 login_events
  → return token

失败计数：
  redis incr login_fail:{username} 60s TTL
  if > 5: 临时锁定 5min（写 users.locked_until）
```

#### 测试要点

- 单元：指纹 hash 稳定、距离计算
- 集成：模拟 5 种场景（新设备 / 同设备 / 异地 / 频繁失败 / 锁定）
- 误报率：人工 review 30 天数据，调阈

#### 依赖

- GeoIP 数据库（本地 ~70MB）
- 邮件 provider
- v1 基础：redis>=5.0

#### 与 v1 的边界

- v1：无异常检测
- v2：告警 + 锁定

#### 风险

- 误报：用户在飞机上 → 接受误报成本（用户可点"是我"按钮解锁）
- 隐私：存 IP → 30 天滚动删除

#### 验收

- [ ] 5 场景全识别
- [ ] 告警邮件到

### 7.2 雪球模型（snowball）增量同步（非 Confluence 通用）

#### 数据流

```
配置: snowball_jobs
  - seed_urls: ["https://wiki.internal/start"]

  - max_depth: 3
  - max_pages_per_run: 1000
  - schedule: "0 2 * * *"

每日 02:00
  → 启 task
  → frontier = [seed]
  → visited = set()
  → for depth in 0..3:
        new_pages = []
        for url in frontier:
            if url in visited: continue
            fetch → parse → upsert
            visited.add(url)
            for link in extract_links(html):
                if link_domain in allowed and link not in visited:
                    new_pages.append(link)
        frontier = new_pages
        if len(visited) > max_pages_per_run: break
```

#### 测试要点

- 单元：BFS 边界、域名白名单、循环检测
